# Supabase Integration Documentation

This document provides detailed information about the Supabase integration in ScholarSift, including database schema, authentication flows, and data management.

## Database Schema

ScholarSift uses Supabase as its primary database and authentication provider. The following tables are defined in the Supabase database:

### Profiles

Stores user profile information and subscription details.

```sql
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id),
  email TEXT NOT NULL,
  subscription_tier TEXT NOT NULL DEFAULT 'free',
  summaries_used INTEGER NOT NULL DEFAULT 0,
  summaries_limit INTEGER NOT NULL DEFAULT 5,
  api_key TEXT,
  zotero_api_key TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Create a trigger to create a profile when a new user signs up
CREATE OR REPLACE FUNCTION public.handle_new_user() 
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO public.profiles (id, email, subscription_tier, summaries_used, summaries_limit)
  VALUES (new.id, new.email, 'free', 0, 5);
  RETURN new;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE PROCEDURE public.handle_new_user();
```

### Summaries

Stores paper summaries generated by users.

```sql
CREATE TABLE summaries (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  user_id UUID NOT NULL REFERENCES profiles(id),
  paper_url TEXT NOT NULL,
  summary_text TEXT NOT NULL,
  date_created TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  title TEXT,
  authors TEXT[],
  publication TEXT,
  year INTEGER
);

-- Create index for faster queries
CREATE INDEX summaries_user_id_idx ON summaries(user_id);
```

### Subscriptions

Stores subscription information for users with paid plans.

```sql
CREATE TABLE subscriptions (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  user_id UUID NOT NULL REFERENCES profiles(id),
  stripe_customer_id TEXT,
  stripe_subscription_id TEXT,
  status TEXT NOT NULL,
  current_period_start TIMESTAMP WITH TIME ZONE,
  current_period_end TIMESTAMP WITH TIME ZONE,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Create index for faster queries
CREATE INDEX subscriptions_user_id_idx ON subscriptions(user_id);
```

## Row Level Security (RLS) Policies

Supabase RLS policies are used to secure data access:

### Profiles Table

```sql
-- Users can only read their own profile
CREATE POLICY "Users can read own profile" ON profiles
  FOR SELECT USING (auth.uid() = id);

-- Users can update their own profile
CREATE POLICY "Users can update own profile" ON profiles
  FOR UPDATE USING (auth.uid() = id);

-- Enable RLS on profiles
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;
```

### Summaries Table

```sql
-- Users can only read their own summaries
CREATE POLICY "Users can read own summaries" ON summaries
  FOR SELECT USING (auth.uid() = user_id);

-- Users can insert their own summaries
CREATE POLICY "Users can insert own summaries" ON summaries
  FOR INSERT WITH CHECK (auth.uid() = user_id);

-- Users can delete their own summaries
CREATE POLICY "Users can delete own summaries" ON summaries
  FOR DELETE USING (auth.uid() = user_id);

-- Enable RLS on summaries
ALTER TABLE summaries ENABLE ROW LEVEL SECURITY;
```

## Authentication Flow

ScholarSift uses Supabase Auth for user authentication. The authentication flow is as follows:

1. **Sign Up**:
   - User provides email and password
   - Supabase creates a new user in the `auth.users` table
   - A trigger automatically creates a profile in the `profiles` table
   - User is assigned the 'free' subscription tier with a limit of 5 summaries

2. **Sign In**:
   - User provides email and password
   - Supabase validates credentials and returns a JWT token
   - The JWT token is stored in local storage for subsequent requests

3. **Password Reset**:
   - User requests a password reset by providing their email
   - Supabase sends a password reset link to the user's email
   - User clicks the link and sets a new password

## API Key Management

API keys are generated for users with Basic or Premium subscription tiers:

1. **API Key Generation**:
   - A secure random string is generated as the API key
   - The API key is stored in the user's profile
   - The API key is prefixed with 'sk-' to indicate it's a secret key

2. **API Key Validation**:
   - When a request is made to the API, the API key is extracted from the Authorization header
   - The API key is validated against the database
   - If valid, the user's profile is retrieved and the request is processed

## Subscription Management

Subscription management is handled through Stripe integration:

1. **Upgrade Subscription**:
   - User selects a subscription tier (Basic or Premium)
   - A Stripe Checkout session is created
   - User completes payment through Stripe
   - Webhook notification is received from Stripe
   - User's profile is updated with the new subscription tier and limits

2. **Downgrade Subscription**:
   - User selects a lower subscription tier
   - Stripe subscription is updated
   - Changes take effect at the end of the current billing period

3. **Cancel Subscription**:
   - User cancels subscription
   - Stripe subscription is cancelled
   - User's profile is updated to 'free' tier at the end of the current billing period

## Data Migration

For migrating from the mock data to Supabase:

1. **Export User Data**:
   - Extract user data from localStorage
   - Format data to match Supabase schema

2. **Import to Supabase**:
   - Create users in Supabase Auth
   - Import profiles and summaries to respective tables

## Error Handling

Errors from Supabase operations are handled as follows:

1. **Authentication Errors**:
   - Display appropriate error messages to the user
   - Log errors for debugging

2. **Database Errors**:
   - Retry operations with exponential backoff
   - Display user-friendly error messages
   - Log detailed error information for debugging

## Security Considerations

1. **API Keys**:
   - API keys are stored securely in the database
   - API keys are never exposed in client-side code
   - API keys can be regenerated by users

2. **User Data**:
   - Sensitive user data is protected by RLS policies
   - Data is encrypted in transit using HTTPS
   - Personal information is minimized
